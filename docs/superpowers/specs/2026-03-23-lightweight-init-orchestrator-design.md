# Yuva AI v3.3.0 — Lightweight Init + Orchestrator

## Problem

1. `yuva init` copies 67 files (~493KB) into every project. Most are never modified by users — they're agent prompts, standards, and checklists that should stay with the package.
2. No auto-detection of AI tools. Users must manually configure for their tool (OpenCode, Codex, Claude Code, Cursor, etc.).
3. Routing logic is embedded as static markdown in CLAUDE.md. The AI may or may not follow it correctly. No existing-code detection happens automatically.

## Solution

### 1. Lightweight Init

**`yuva init` creates 3-4 files instead of 67:**

```
project/
├── AGENTS.md                          # Main orchestrator (replaces CLAUDE.md)
├── .aiautomations/
│   ├── config.json                    # Tool config + package install path
│   └── agents.md                      # Index of all 11 development agents with descriptions
└── [tool-specific pointer file]       # Only if needed
```

Agent prompts, standards, checklists, protocols, and templates remain in the installed npm package. They are accessed on demand via `yuva agent show <name>`.

**CLAUDE.md is renamed to AGENTS.md** — a universal name not tied to any specific AI vendor.

### 2. On-Demand Agent Resolver

New CLI commands:

```
yuva agent show <name>       # Reads agent prompt from package, outputs to stdout
yuva agent list              # Lists all available agents from package
yuva agent orchestrate       # Scans project context, returns JSON
```

**`yuva agent show <name>`:**
- Resolves the package install path from `.aiautomations/config.json`
- Falls back to: global npm path → local node_modules → bundled path
- Reads `template/.aiautomations/prompts/<name>agent.md` from package
- Outputs full prompt to stdout
- If a local override exists in `.aiautomations/prompts/<name>agent.md`, uses that instead (backward compat)

**`yuva agent orchestrate`:**
- Scans the project directory for context:
  - Existing code? (checks for src/, lib/, package.json, requirements.txt, go.mod, etc.)
  - What framework/language? (detects from files)
  - Git status? (clean, dirty, uninitialized)
  - Session files? (.session/state.md exists = resumable project)
  - Which AI tool is active? (from config.json)
- Returns structured JSON:

```json
{
  "hasExistingCode": true,
  "languages": ["javascript", "typescript"],
  "frameworks": ["express", "react"],
  "hasSession": false,
  "gitStatus": "clean",
  "tool": "opencode",
  "suggestedFirstAgent": "existingcode"
}
```

### 3. AI Tool Auto-Detection

`yuva init` auto-detects the AI tool:

**Detection priority:**
1. Environment variables: `CURSOR_*`, `VSCODE_*`, `TERM_PROGRAM`
2. Running inside known IDE terminal (check parent process)
3. Existing config files: `.cursor/`, `.github/copilot-instructions.md`, etc.
4. Installed CLIs: `which claude`, `which opencode`, `which codex`, `which aider`
5. Fallback: create generic AGENTS.md

**Manual override:** `yuva init <tool>` (e.g., `yuva init opencode`, `yuva init cursor`)

**Switch later:** `yuva llm use <tool>` continues to work.

### 4. Tool-Specific Pointer Files

For tools that require their own config file location:

| Tool | Pointer File | Points To |
|------|-------------|-----------|
| Cursor | `.cursor/rules/yuva.mdc` | AGENTS.md |
| Copilot | `.github/copilot-instructions.md` | AGENTS.md |
| Windsurf | `.windsurfrules` | AGENTS.md |
| Kilo Code | `.kilo/instructions.md` | AGENTS.md |
| Cody | `.sourcegraph/instructions.md` | AGENTS.md |
| Amazon Q | `.amazonq/instructions.md` | AGENTS.md |

Pointer file content:
```markdown
# Yuva AI
<!-- Auto-generated by yuva init. Source of truth: AGENTS.md -->
Read and follow all instructions in the AGENTS.md file in the project root.
All agent prompts are available via: yuva agent show <agent-name>
```

Tools that already read AGENTS.md natively (OpenCode, Codex, GPT) need no pointer file.

### 5. Hybrid Orchestrator

The orchestrator is split between CLI (deterministic checks) and AI (intent understanding).

**AGENTS.md content (simplified):**

```markdown
# Yuva AI

You have access to a multi-agent system with 11+ specialized development agents.

## How to handle any request

1. Run `yuva agent orchestrate` to scan project context
2. Read the JSON output to understand the project state
3. Based on the user's intent + project context, select the right agent(s)
4. Run `yuva agent show <name>` to get each agent's full instructions
5. Follow agent instructions in order

## Agent Selection Guide

### If existing code detected (hasExistingCode: true)
→ Always run `yuva agent show existingcode` FIRST

### Development tasks
- "build/create" → requirements → planner → execution
- "fix/debug/error" → debugger
- "test/QA" → tester
- "review/audit" → reviewer
- "refactor/clean" → refactor
- "security" → security
- "continue/resume" (hasSession: true) → continuity

## Available agents
Run `yuva agent list` to see all agents with descriptions.
```

**Flow:**
```
User message
  → AI reads AGENTS.md
  → AI runs `yuva agent orchestrate` (CLI scans project)
  → CLI returns JSON context
  → AI uses context + user intent to pick agents
  → If hasExistingCode → existingcodeagent runs first
  → AI runs `yuva agent show <name>` for each selected agent
  → Follows instructions in order
```

### 6. Backward Compatibility

- Projects with old full-copy setup continue to work — local files in `.aiautomations/prompts/` take priority over package files
- `yuva upgrade` migrates: renames CLAUDE.md → AGENTS.md, removes copied prompts that match package versions, creates config.json
- `yuva doctor` validates both old and new setups

### 7. config.json Schema

```json
{
  "tool": "opencode",

  "version": "3.3.0",
  "autoDetected": true,
  "telemetry": false,
  "sessionPersistence": true
}
```

## Testing

- `yuva init` creates exactly 3-4 files
- `yuva init opencode` creates AGENTS.md + OpenCode-compatible config
- `yuva init cursor` creates AGENTS.md + `.cursor/rules/yuva.mdc` pointer
- `yuva agent show requirements` outputs full requirementsagent.md content
